---

## 1. Purpose of this report

This report describes the product-level data structure generated by the S8 release stage of the `sed_data_integration` workflow.

The S8 stage is designed as a publication and packaging layer. It does not rerun the upstream data harmonization, quality control, spatial clustering, basin matching, or source-merging workflow. Instead, it collects the already generated S6/S7 products and repackages them into a user-facing release directory with standardized file names, NetCDF products, tabular catalogs, GIS sidecars, provenance sidecars, README documentation, and validation reports.

The resulting release product is not a single NetCDF file. It is a multi-file, relational, provenance-aware data package designed for:

- global river suspended sediment reference data publication;
- model-observation comparison;
- station and basin based spatial selection;
- time-series extraction at daily, monthly, and annual resolutions;
- record-level and source-station-level provenance tracing;
- overlap diagnostics among multiple source datasets;
- separate handling of climatology and satellite validation-only observations.

## 3. Product-level design

### 3.1 Main design concept

The S8 product is organized around three linked data layers:

```text
Cluster / virtual station layer
        │
        ├── Source-station provenance layer
        │
        └── Observation record or station-time cell layer
```

The mainline station reference product is based on spatially merged clusters. Each cluster represents a virtual reference station assembled from one or more original source stations. Each final observation record or matrix cell is linked back to the selected original source station.

### 3.2 Core join keys

The most important join keys are:

| Key | Meaning | Main use |
|---|---|---|
| `cluster_uid` | Stable ID for a merged cluster or virtual station. Usually formatted as `SED######`. | Primary cross-file join key for station, matrix, catalog, and GIS products. |
| `cluster_id` | Numeric cluster identifier from the upstream merge workflow. | Internal numeric index and compatibility field. |
| `source_station_uid` | Stable ID for an original source station. Usually formatted as `SRC######`. | Provenance key for linking selected records or matrix cells back to source-station metadata. |
| `resolution` | Temporal resolution of the record or product. | Distinguishes daily, monthly, annual, and climatology products. |
| `time` | Numeric time coordinate, generally encoded as days since 1970-01-01. | Time-series alignment and model-observation comparison. |
| `station_index` | NetCDF internal index pointing to `n_stations`. | Used inside long-table NetCDF products. |
| `source_station_index` | NetCDF internal index pointing to `n_source_stations`. | Record-level provenance link in `sed_reference_master.nc`. |
| `selected_source_station_uid` | Cell-level provenance key in matrix products. | Allows matrix users to trace each selected value back to the source station. |

### 3.3 Temporal resolutions

The mainline matrix products are separated by temporal resolution:

| Resolution | Product file |
|---|---|
| Daily | `sed_reference_timeseries_daily.nc` |
| Monthly | `sed_reference_timeseries_monthly.nc` |
| Annual | `sed_reference_timeseries_annual.nc` |

Climatology and satellite validation-only data are handled separately:

| Data type | Product file | Role |
|---|---|---|
| Climatology | `sed_reference_climatology.nc` | Standalone climatology product, not basin-merged into the mainline station product. |
| Satellite validation-only data | `sed_reference_satellite.nc` | Validation-only product, excluded from the main station-reference merge. |

## 4. Core NetCDF products

## 4.1 `sed_reference_master.nc`

### 4.1.1 Role

`sed_reference_master.nc` is the authoritative long-table archive of the release product. It stores selected or winning records after source merging and provides the most complete record-level provenance structure.

It is the best entry point when users need:

- complete record-level metadata;
- source-station provenance;
- selected source information;
- overlap flag information;
- linkage between station clusters and original source stations.

### 4.1.2 Main dimensions

```text
n_stations
n_source_stations
n_records
n_sources
```

| Dimension | Meaning |
|---|---|
| `n_stations` | Number of merged clusters or virtual stations. |
| `n_source_stations` | Number of original source-station entries mapped to the merged clusters. |
| `n_records` | Number of selected long-table observation records. |
| `n_sources` | Number of source datasets represented in the release product. |

### 4.1.3 `n_stations` layer

The `n_stations` layer describes the merged cluster or virtual station.

Representative variables include:

```text
lat
lon
cluster_id
cluster_uid
station_name
river_name
source_station_id
sources_used
n_source_stations_in_cluster
basin_area
pfaf_code
n_upstream_reaches
basin_match_quality
basin_distance_m
point_in_local
point_in_basin
basin_status
basin_flag
```

Important fields:

| Variable | Description |
|---|---|
| `cluster_uid` | Stable cluster identifier, typically `SED######`. This is the preferred join key across NetCDF, CSV, and GPKG products. |
| `cluster_id` | Numeric cluster ID. |
| `lat`, `lon` | Representative station or cluster coordinate. |
| `station_name`, `river_name` | Representative station and river names. |
| `sources_used` | Source datasets contributing to the cluster. |
| `n_source_stations_in_cluster` | Number of original source stations mapped to the cluster. |
| `basin_area` | Matched basin area. |
| `pfaf_code` | Pfafstetter basin code if available. |
| `basin_match_quality` | Basin matching quality indicator. |
| `basin_distance_m` | Distance between station point and matched basin geometry or outlet/reference geometry. |
| `point_in_local`, `point_in_basin` | Spatial relationship flags for station point and basin geometry. |
| `basin_status`, `basin_flag` | Basin matching status and quality flag. |

### 4.1.4 `n_source_stations` layer

The `n_source_stations` layer stores original source-station provenance information.

Representative variables include:

```text

source_station_data_limitations
source_station_declared_temporal_resolution
```

Important fields:

| Variable | Description |
|---|---|
| `source_station_uid` | Stable source-station identifier, typically `SRC######`. |
| `source_station_cluster_index` | Link from source station to the merged cluster. |
| `source_station_source_index` | Link from source station to the source dataset lookup table. |
| `source_station_native_id` | Original station ID from the source dataset. |
| `source_station_name` | Original station name. |
| `source_station_river_name` | Original river name. |
| `source_station_lat`, `source_station_lon` | Original source-station coordinates. |
| `source_station_paths` | Original or organized source NetCDF file paths. |
| `source_station_resolutions` | Temporal resolutions provided by the source station. |
| `source_station_temporal_span` | Source-station temporal coverage description. |
| `source_station_variables_provided` | Variables provided by the source station. |
| `source_station_data_limitations` | Source-specific limitations or caveats. |

Each selected observation record in the `n_records` layer can be traced back to this layer through `source_station_index`.

### 4.1.5 `n_sources` layer

The `n_sources` layer is a source-dataset lookup table.

Representative variables include:

```text
source_name
source_long_name
institution
reference
source_url
```

| Variable | Description |
|---|---|
| `source_name` | Short source dataset name. |
| `source_long_name` | Full source dataset name. |
| `institution` | Data provider or institution. |
| `reference` | Citation or reference information. |
| `source_url` | Source dataset URL if available. |

### 4.1.6 `n_records` layer

The `n_records` layer is the selected observation long table.

Representative variables include:

```text
station_index
source_station_index
resolution
time
Q
SSC
SSL
Q_flag
SSC_flag
SSL_flag
is_overlap
source
```

| Variable | Description |
|---|---|
| `station_index` | Internal index linking the record to `n_stations`. |
| `source_station_index` | Internal index linking the record to `n_source_stations`. |
| `resolution` | Temporal resolution code. |
| `time` | Time coordinate, generally days since 1970-01-01. |
| `Q` | River discharge. |
| `SSC` | Suspended sediment concentration. |
| `SSL` | Suspended sediment load. |
| `Q_flag`, `SSC_flag`, `SSL_flag` | Quality flags for each variable. |
| `is_overlap` | Flag indicating whether multiple source records competed for the same cluster-resolution-time key. |
| `source` | Source dataset index or code. |

### 4.1.7 Main scientific variables

| Variable | Meaning | Typical unit |
|---|---|---|
| `Q` | River discharge | `m3 s-1` |
| `SSC` | Suspended sediment concentration | `mg L-1` |
| `SSL` | Suspended sediment load | `ton day-1` |

### 4.1.8 Quality flags

The product uses standardized variable-level quality flags:

| Flag | Meaning |
|---|---|
| `0` | Good |
| `1` | Estimated |
| `2` | Suspect |
| `3` | Bad |
| `9` | Missing |

These flags are stored separately for `Q`, `SSC`, and `SSL`.

### 4.1.9 Overlap flag

The variable `is_overlap` indicates whether multiple source records were available for the same merged target key:

```text
(cluster_id, resolution, date)
```

If `is_overlap = 1`, multiple source records competed at that time step, and the final value stored in `sed_reference_master.nc` is the selected or winning record after source ranking and quality-based selection.

Important note:

`sed_reference_master.nc` stores only the selected record. It does not store all unselected candidate records. For full overlap diagnostics, use:

```text
sed_reference_overlap_candidates.csv.gz
```

- quick provenance tracing using `selected_source_station_uid`.

## 4.5 Shared matrix-product design

The three matrix products share the same core logic.

### 4.5.1 Matrix layout

```text
Rows    = stations / clusters
Columns = shared time steps for one temporal resolution
Cells   = selected observation values
```

### 4.5.2 Matrix product advantages

Compared with the master long-table file, matrix products are easier to use for:

- model validation;
- time-series extraction;
- gridded model-to-station comparison;
- station screening by valid time steps;
- direct plotting;
- array-based statistical computation.

### 4.5.3 Matrix provenance

Each populated matrix cell has a selected source-station provenance key:

```text
selected_source_station_uid
```

This key can be joined to:

```text
source_station_catalog.csv
```

or to the source-station layer in:

```text
sed_reference_master.nc
```

### 4.5.4 Release filtering rule

Matrix cells are populated only when at least one sediment variable is available:

```text
SSC is not missing
or
SSL is not missing
```

Q-only cells are not exported as sediment-reference observations in the release product.

## 6. Overlap candidate sidecar

## 6.1 `sed_reference_overlap_candidates.csv.gz`

### 6.1.1 Role

`sed_reference_overlap_candidates.csv.gz` is a candidate-level provenance sidecar.

It is used when multiple source records are available for the same target key:


### 6.1.3 Main use

This file is required for:

### 6.1.4 Important design note

`is_overlap = 1` in `sed_reference_master.nc` or the matrix files only indicates that an overlap existed. It does not provide the unselected candidate values.

To recover all candidate values, users must use:


## 9. Recommended user workflows

## 9.1 Model-observation comparison workflow

Recommended workflow for comparing model outputs with the reference dataset:

1. Identify the temporal resolution of the model output.

```text
daily model output   -> sed_reference_timeseries_daily.nc
monthly model output -> sed_reference_timeseries_monthly.nc
annual model output  -> sed_reference_timeseries_annual.nc
```

2. Use `station_catalog.csv` or `sed_reference_cluster_points.gpkg` to select stations.

3. Use `cluster_uid` to locate the corresponding row in the matrix product.

4. Extract observed `Q`, `SSC`, or `SSL`.

5. Align the model and observation time axes.

6. Apply quality-flag filtering if needed.

7. Compute model performance metrics.

8. Use `selected_source_station_uid` to document the source-station provenance of the selected observations.

---

## 9.2 Provenance tracing workflow

Recommended workflow for tracing a matrix value back to the original source station:

1. Open one of the matrix files:

```text
sed_reference_timeseries_daily.nc
sed_reference_timeseries_monthly.nc
sed_reference_timeseries_annual.nc
```

2. Select a station and time cell.

3. Read:

```text
cluster_uid
time
resolution
selected_source_station_uid
```

4. Join `selected_source_station_uid` to:

```text
source_station_catalog.csv
```

5. Retrieve source station metadata:

```text
source_name
source_station_native_id
source_station_name
source_station_river_name
source_station_lat
source_station_lon
source_station_paths
reference
source_url
```

6. For record-level details, query `sed_reference_master.nc` using:

```text
cluster_uid + resolution + time
```

---

## 9.3 Overlap diagnostic workflow

Recommended workflow for analyzing multi-source overlap:

1. Identify records or cells with:

```text
is_overlap = 1
```

2. Open:

```text
sed_reference_overlap_candidates.csv.gz
```

3. Filter by:

```text
cluster_uid
resolution
date
```

4. Compare all candidate records:

```text
source
source_station_uid
candidate_rank
candidate_quality_score
selected_flag
Q
SSC
SSL
Q_flag
SSC_flag
SSL_flag
```

5. Use `selected_flag = 1` to identify the record published in master and matrix products.

6. Use non-selected candidates to quantify source disagreement or uncertainty.

---

## 9.4 GIS workflow

Recommended workflow for spatial selection:

1. Open:

```text
sed_reference_cluster_points.gpkg
```

2. Select the appropriate layer:

```text
cluster_daily
cluster_monthly
cluster_annual
```

3. Filter by region, basin, country, or model domain.

4. Export selected `cluster_uid` values.

5. Use the selected `cluster_uid` values to extract time series from the corresponding matrix NetCDF product.

6. Optionally join to:

```text
station_catalog.csv
```

for record counts, time coverage, source information, and basin attributes.

## 10. Important release design rules

## 10.1 Sediment-only release filtering

The mainline release product only publishes records or matrix cells where at least one sediment variable is available:

```text
SSC is not missing
or
SSL is not missing
```

Q-only time steps are not included as sediment-reference records.

This design ensures that the release product remains focused on suspended sediment reference data.

---

## 10.2 Satellite exclusion from the main merge

Satellite and satellite-like records are excluded from the main station-reference merge.

They are published separately in:

```text
sed_reference_satellite.nc
satellite_catalog.csv
```

This prevents validation-only satellite observations from being mixed with the station-based reference product.

---

## 10.3 Climatology separation

Climatology records are published separately in:

```text
sed_reference_climatology.nc
```

They are not merged into the daily, monthly, or annual basin-clustered mainline products.

---

## 10.4 Selected-record-only master and matrix products

The following products store only selected or winning records:

```text
sed_reference_master.nc
sed_reference_timeseries_daily.nc
sed_reference_timeseries_monthly.nc
sed_reference_timeseries_annual.nc
```

They do not store unselected overlap candidates.

Full selected and non-selected candidate information is stored in:

```text
sed_reference_overlap_candidates.csv.gz
```

---

## 10.5 No automatic temporal aggregation of model outputs

The release package provides observation products at daily, monthly, and annual resolutions. It does not automatically aggregate model outputs from one resolution to another.

Users should ensure that model outputs and reference observations are compared at consistent temporal resolutions.

## 11. Recommended citation and documentation language

For manuscript or data descriptor writing, the S8 release product can be described as follows:

> The final release product is distributed as a multi-file, provenance-aware reference dataset. The package includes an authoritative long-table NetCDF archive, three resolution-specific station-by-time matrix NetCDF files for daily, monthly, and annual model evaluation, standalone climatology and satellite validation-only NetCDF products, tabular station and source catalogs, GIS sidecars, an overlap-candidate provenance table, and release-level validation reports. The core connection keys are `cluster_uid`, `source_station_uid`, `resolution`, and `time`, which together support station selection, time-series extraction, source tracing, and overlap diagnostics.

A shorter version is:

> The S8 release package organizes the sediment reference dataset as a relational data product composed of NetCDF observations, CSV provenance catalogs, GIS sidecars, and validation reports. The matrix NetCDF files provide model-ready station-by-time observations, while the master NetCDF and source catalogs preserve record-level provenance.

The S8 release product is a publication-level data package rather than a single data file.

Its core components are:

```text

    Authoritative long-table archive with selected records and record-level provenance.

sed_reference_timeseries_daily.nc
sed_reference_timeseries_monthly.nc
sed_reference_timeseries_annual.nc
    Model-ready station-by-time matrix products.

    Validation-only satellite product.

station_catalog.csv
    Cluster/resolution-level station catalog.

source_station_catalog.csv
    Source-station/resolution-level provenance catalog.

source_dataset_catalog.csv
    Source-dataset-level metadata catalog.

satellite_catalog.csv
    Satellite validation-only station catalog.

sed_reference_overlap_candidates.csv.gz
    Candidate-level overlap provenance table.

sed_reference_cluster_points.gpkg
sed_reference_source_stations.gpkg
sed_reference_cluster_basins.gpkg
    GIS sidecars for spatial search, mapping, and basin visualization.

release_validation_report.csv
release_inventory.csv
    Release validation and inventory files.
```

The most important user-facing keys are:

```text
cluster_uid
resolution
time
source_station_uid
```

For model validation, the recommended entry points are:

```text
sed_reference_timeseries_daily.nc
sed_reference_timeseries_monthly.nc
sed_reference_timeseries_annual.nc
```

For provenance tracing, the recommended entry points are:

```text
sed_reference_master.nc
source_station_catalog.csv
sed_reference_overlap_candidates.csv.gz
```

In summary, the S8 release package is a model-evaluation-ready, spatially indexed, provenance-preserving global river suspended sediment reference dataset.
